## 2. New Architecture

```
User's Browser
    │
    ▼
PythonAnywhere (Flask app)
    │
    ├── / ........................ Landing page (links to form, calendar, email, ICS)
    │
    ├── /login .................. Enter email → receive magic link
    ├── /auth/<token> ........... Verify magic link → set session
    ├── /logout ................. Clear session
    │
    ├── /form ................... Trade/comment submission form (auth required)
    ├── /calendar ............... Year-at-a-glance view (public, read-only)
    ├── /calendar/<year> ........ Specific year view (public, read-only)
    ├── /ics .................... ICS download form (public)
    ├── /ics/download ........... Generate and serve ICS file
    ├── /email .................. Compose URL page (auth required)
    │
    ├── /admin .................. Admin dashboard (Larry only)
    │   ├── /admin/owners ....... Manage owner list
    │   ├── /admin/audit ........ View audit log
    │   └── /admin/export ....... Export data to Excel
    │
    ▼
SQLite database (snowbound.db)
    │
    ├── owners
    ├── owner_emails
    ├── calendar
    ├── mud_weeks
    ├── trade_detail
    ├── audit
    └── magic_links
```

### Why SQLite (not MySQL)

- Data volume is tiny (~4,000 rows in TradeDetail, ~800 in Calendar, 10 owners)
- Single-user writes (only one person submits a form at a time)
- No need for a separate database server
- File-based — easy to back up (just copy the .db file)
- PythonAnywhere supports it on all tiers including free
- Can always migrate to MySQL/Postgres later if needed

## 4. Authentication Flow (Magic Link)

```
1. User visits /login
2. User enters their email address
3. Server checks email exists in owner_emails table
4. If yes: generate token, store in magic_links, send email with link
5. If no: show generic "If this email is registered, you'll receive a link"
6. User clicks link in email → /auth/<token>
7. Server validates token (not expired, not used, marks as used)
8. Server sets Flask session with owner_id
9. User is redirected to /form
```

### Session Duration
- Session lasts 7 days (configurable)
- Owners don't use this daily, so make it long enough they don't have to re-auth every visit
- Simple "Remember me" cookie

### Email Sending for Magic Links
- Use SMTP through the 69hawn.com domain (or Gmail app password)
- Only sends ~10-20 emails/month — trivial volume
- Since it's transactional (one email, one recipient, one link), DMARC is straightforward
- Alternative: use PythonAnywhere's built-in SMTP if available on paid tier

### Public vs Auth-Required Routes

| Route | Auth Required? | Notes |
|---|---|---|
| /login | No | |
| /calendar, /calendar/YYYY | No | Read-only views are public |
| /ics, /ics/download | No | Anyone can download a calendar file |
| /form | Yes | Must be logged in to submit trades |
| /email | Yes | Must be logged in to compose group email |
| /admin/* | Yes + is_admin | Larry only |

## 5. Key Application Logic

### Form Processing (replaces processForm + ProcessUnprocessed)

```
POST /form
1. Validate session (who is logged in)
2. Validate form data:
   - trade_type is valid enum
   - owner1 exists
   - owner1_week exists in trade_detail for that owner
   - If "Trade Week": owner2 and owner2_week also required
   - comment ≤ 40 chars
3. Process based on trade_type:
   - "Trade Week": swap current_holder on both rows in trade_detail
   - "Give Away": update current_holder on owner1's row
   - "Not Using": add comment to owner1's row
   - "Comment": add comment only
4. Write audit record
5. Return success/error to user
```

### Calendar View (replaces PreviousYear/CurrentYear/NextYear sheets)

```
GET /calendar         → shows current year
GET /calendar/2025    → shows specific year
GET /calendar/prev    → shows previous year
GET /calendar/next    → shows next year

Query: SELECT td.*, o.display_info
       FROM trade_detail td
       JOIN owners o ON td.owner_id = o.id
       WHERE td.year = :year
       ORDER BY td.week_start

Template renders the same grid as the Google Sheet year views:
- Owner info | Week 1 | Week 2 | Week 3 | Week 4 | Week 5
- With trade comments inline
- Include garage code, WiFi, lock box at top (store in a config table or app config)
```

### ICS Generation (replaces FormICS + handleICSSubmission)

```
GET  /ics              → form: select owner + year
POST /ics/download     → generate and return .ics file

Query owner's weeks for the given year from trade_detail
Build ICS content (VCALENDAR/VEVENT blocks) in Python
Return as file download with content-type text/calendar
```

### Compose Email (replaces Email.html + getComposeUrl)

```
GET /email → page with Gmail/Outlook/Copy buttons

Query all emails from owner_emails table
Generate compose URLs client-side (same JS logic you already have)
No server-side email sending needed
```

## 6. Conversion Strategy

### Phase 1: Set Up Infrastructure

- [ ] Create PythonAnywhere account (free tier to start)
- [ ] Set up Flask app skeleton with SQLAlchemy
- [ ] Create SQLite database with schema above
- [ ] Set up basic Flask-Login or session management

### Phase 2: Seed the Database

- [ ] **owners + owner_emails**: Manual entry from NameList (only 10 owners)
- [ ] **calendar**: Run the Python rotation generator (already written in generateCondoCalendar.js comments) to populate calendar table from 2022-2100
- [ ] **mud_weeks**: Generated by the same script
- [ ] **trade_detail**: 
  - Export TradeDetail sheet to CSV
  - Write a one-time Python import script
  - Map owner names to owner_ids
  - Preserve all existing trade history, comments, and audit trails
- [ ] **audit**: Export Audit sheet to CSV, import similarly
- [ ] Verify row counts match between sheets and database

### Phase 3: Build Core Routes

- [ ] /calendar (read-only year view) — easiest to build, easiest to verify
- [ ] /ics (calendar download) — self-contained, easy to test
- [ ] /login + /auth (magic link) — needed before form
- [ ] /form (trade submission) — most complex, test thoroughly
- [ ] /email (compose URLs) — straightforward port of existing JS

### Phase 4: Parallel Testing

- [ ] Run both GAS app and Flask app simultaneously
- [ ] Have Larry test all form operations on Flask version
- [ ] Compare trade_detail table with Google Sheet after each test
- [ ] Verify ICS output matches between old and new

### Phase 5: Cutover

- [ ] Final data sync from Google Sheet → SQLite
- [ ] Point users to new URL
- [ ] Keep GAS app alive (read-only) for 30 days as fallback
- [ ] Decommission GAS app

### Phase 6: Post-Migration Enhancements (Optional)

- [ ] Admin page for Larry to manage owners without touching the database
- [ ] Export-to-Excel button for familiar spreadsheet view of data
- [ ] Mobile-friendly CSS (elderly users on iPads/phones)
- [ ] Automated daily backup of SQLite file

## 9. Risk Mitigation

| Risk | Mitigation |
|---|---|
| Data loss during migration | Export all sheets to CSV before starting; keep Google Sheet as read-only backup |
| Date format issues | SQLite stores dates as TEXT in MM/DD/YYYY (matching existing format); validate on import |
| Elderly users confused by change | Magic link is simpler than current flow; calendar views look the same; send a one-page guide |
| PythonAnywhere goes down | Daily automated backup of SQLite file to Google Drive or email |
| Need to go back | Google Sheet + GAS app remains functional for 30-day overlap period |

## 10. What Gets Simpler

- **No more multi-account browser bug** — it's a standalone website
- **No more "all cells must be plain text" constraint** — proper database types
- **No more cache hacks** (CacheService, nukeWeeksCacheFast) — database queries are fast
- **No more Mailjet API keys in source code** — compose URLs only, SMTP config in .env
- **No more GAS deployment versioning confusion** — push code, it's live
- **No more anonymous access workarounds** — magic link auth is clean and simple
- **Date handling** — Python `datetime` is far more reliable than Google Sheets text-formatted dates
